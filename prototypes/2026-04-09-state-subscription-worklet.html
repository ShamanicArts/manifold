Memory Ordering Semantics

// Audio thread (producer) void write(Notification note) { buffer_[writeIdx] = note; // Write data first writeIdx.store((writeIdx + 1) % SIZE, std::memory_order_release); // Publish with release } // Message thread (consumer) bool read(Notification& note) { size_t w = writeIdx.load(std::memory_order_acquire); // Acquire to see writes size_t r = readIdx.load(std::memory_order_relaxed); if (r == w) return false; // Empty note = buffer_[r]; // Read data after acquire readIdx.store((r + 1) % SIZE, std::memory_order_release); // Publish read with release return true; }

Key insight: The release-acquire pair ensures the audio thread's data write is visible to the message thread before the index update is seen. This matches the pattern used in the user's SPSCQueuePtr implementation.

Integration with BehaviorCoreProcessor

class BehaviorCoreProcessor : public juce::AudioProcessor { // ... existing members ... // New: State subscription system struct StateChange { uint32_t pathId; float value; uint64_t timestamp; }; // Lock-free queue for audio-to-message thread communication SPSCQueue<StateChange, 256> pendingChanges_; // Message-thread only: subscriber callbacks std::unordered_map<uint32_t, std::vector<Callback>> subscribers_; std::mutex subscribersMutex_; // Only accessed on message thread public: // Called from audio callback - MUST be lock-free void notifyStateChange(const std::string& path, float value) { uint32_t id = getPathId(path); // Pre-computed hash pendingChanges_.enqueue({id, value, sampleClock_}); } // Called from message thread timer void processPendingChanges() { StateChange change; while (pendingChanges_.try_dequeue(change)) { auto it = subscribers_.find(change.pathId); if (it != subscribers_.end()) { for (auto& cb : it->second) { cb(change.value); } } } } // Called from WebSocket or UI thread void subscribeToPath(const std::string& path, Callback cb) { uint32_t id = getPathId(path); subscribers_[id].push_back(cb); } };

Key Design Decisions

Aspect Decision Rationale
Queue size 256 entries Enough for 5ms at 48kHz with one change per sample
Path identification Pre-hashed uint32_t O(1) lookup, no string hashing in audio thread
Overflow handling Drop oldest (circular) Message thread should keep up; if not, stale data is less valuable
Subscriber storage Hash map + vectors Message thread only - can use standard containers
Timestamp Sample clock (uint64_t) Deterministic ordering, correlates with audio
